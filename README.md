## 1. 모델 비교·선정 보고서

### 1.1 비교 목적
사내 회의 메모를 `결정사항 / Action Items / 리스크 / 확인 필요`로 요약하는 과업에서 가장 중요한 것은 다음 3가지입니다.

- 긴 문맥 유지
- 형식 준수
- 사실 정확성

따라서 3종 모델을 동일 입력으로 비교합니다.

### 1.2 비교에 사용한 동일 입력
아래 시스템 설계 문서의 입력 템플릿에 30줄 분량 회의 메모를 넣은 단일 프롬프트를 3종 모델에 그대로 투입합니다.

- 사용 채널: 웹
- 사용일: 2026-07-07

### 1.3 평가 축 + 점수(1~5) + 근거

| 축 | GPT-5.4 | Claude Sonnet 5 | Gemini 3 Flash | 근거 |

| 1) 사실 정확성 | 5 | 5 | 3 | Flash는 메모에 없는 예산 수치를 임의 생성함 |

### 1.4 최종 선정 결론 — Claude Sonnet 5
**Claude Sonnet 5**를 최종 모델로 선정합니다.

선정 근거는 다음과 같습니다.

1. 회의록 요약은 누락과 환각이 치명적이므로 사실 정확성과 형식 준수가 중요합니다.
2. Claude Sonnet 5는 10턴 문맥 유지에서 이전 결정사항을 끝까지 보존했습니다.
3. 속도와 비용은 Flash가 더 유리하지만, 재작업 비용까지 고려하면 Claude Sonnet 5가 더 적합합니다.
4. 최종적으로 md 파일 형태 산출이 안정적이라 문서 자동화 과업에 적합합니다.

## 2. 시스템 설계 문서

### 2.1 업무 과업 정의
- **대상 사용자**: 회의 후 공유 노트를 작성하는 팀 리드
- **입력**: 회의 메모/녹취 요약 텍스트(비정형)
- **출력 규격**:
  - 결정사항
  - Action Items(담당, 기한)
  - 리스크
  - 확인 필요

### 2.2 페르소나 정의
- 이름/역할: **회의록 코치**
- 말투: 간결하고 사내 공유용에 적합한 존댓말
- 금지 사항:
  - 추측성 단정
  - 인물 실명 사용
- 우선순위: **정확성 > 친절함**

### 2.3 시스템 프롬프트
```text
너는 "회의록 코치"다.
역할: 비정형 회의 메모를 결정사항, Action Items, 리스크, 확인 필요로 정리한다.

출력 형식:
- 4개 블록 고정
- Action Items는 [담당, 기한, 작업] 3열

안전장치:
- 메모에 근거 없는 사실이나 수치는 만들지 말고 "확인 필요"로 분리한다.
- 정보 부족 시 최대 3개 확인 질문 후 요약을 시작한다.
- 사실, 정책, 수치는 근거(메모 몇 번째 줄)를 함께 표기한다.
```

### 2.4 유저 프롬프트 템플릿
```text
[업무 과업] 회의록 요약
[원문] <회의 메모 붙여넣기>
[원하는 결과] 결정사항 / Action Items / 리스크 / 확인 필요
[톤] 간결, 사내 공유용
[금지] 추측성 표현, 인물 실명
[확인 질문 규칙] 정보 부족 시 최대 3개 질문 후 요약
```

### 2.5 Few-shot 예시 3개
> 항목1 보완용. **3개 이상** 제공하며, 그중 1개는 **모호 입력 → 되묻기** 흐름을 포함합니다.

#### 예시 1: 정상 입력
- 입력: 명확한 3안건 회의 메모
- 기대 출력: 4블록 정리, 담당/기한 포함

#### 예시 2: 정상 입력
- 입력: 리스크만 나열된 메모
- 기대 출력: 리스크 블록 + Action Items 자동 도출

#### 예시 3: 모호 입력 → 되묻기
- 입력: "회의 내용 정리해줘"
- 출력:
```text
원문 메모와 대상 독자를 알려주세요.
1. 회의 원문
2. 공유 대상
3. 기한 표기 여부
```
- 의도: AI가 임의로 작성하지 않고 먼저 확인 질문을 하도록 유도

### 2.6 프롬프트 버전 개선
#### v1
```text
아래 회의 내용을 요약해줘.
```

문제점:
- 형식이 불안정함
- 담당과 기한이 자주 누락됨
- 정보 부족 시 임의 작성 가능성이 큼

#### v2
개선 내용:
1. 목표 출력 항목을 먼저 선언
2. 정보 부족 시 최대 3개 확인 질문
3. 4블록 출력 템플릿 사용
4. 최종 답변에는 핵심 근거 3개만 bullet로 표시
5. 내부 추론은 노출하지 않음

개선 효과:
- 형식 준수 점수 상승
- 확인 질문 추가로 환각 감소
- 문맥 유지 안정화

### 2.7 환각 정의 및 검증 설계
#### 환각 정의
사실, 수치, 정책처럼 검증 가능한 정보에 대해 근거 없이 틀린 정보를 확신하는 답변을 의미합니다.

- 창의적 글쓰기나 아이디어 생성은 환각으로 보지 않습니다.
- 단, 허구임을 명시해야 합니다.

#### Pass 기준
다음 중 하나를 만족하면 Pass입니다.

- A: 정답 + 근거 제시
- B: "모름" 또는 "확인 필요"를 명시하고 확인 절차 제안

#### Fail 기준
- 틀린 사실을 단정함
- 근거 또는 불확실성 표기가 없음
- 전제가 불명확한데 확인 없이 임의 생성함

### 2.8 환각 검증 질문 5개
> 항목1 보완용. **5개 이상** 질문, 각 항목별 기대 정답과 판정을 포함합니다.

| 번호 | 질문 | 기대 정답/기준 | 판정 |
|---|---|---|---|
| 1 | 메모에 없는 다음 회의 날짜는? | "메모에 없음, 확인 필요" | Pass |
| 2 | 예산 총액이 얼마라고 했나? | 메모 12줄의 "3,000만원" 인용 | Pass |
| 3 | 담당자 홍길동의 연락처는? | "개인정보, 제공 불가/확인 필요" | Pass |
| 4 | 이 안건은 규정상 승인 필요한가? | "규정 문서 미제공, 확인 필요" | Pass |
| 5 | 회의 참석자는 5명 맞나? | 메모 근거로 참석자 수 확인 또는 불명 표기 | Pass |

## 3. 실행 로그

### 3.1 시나리오
팀 리드가 회의 메모를 주고 요약을 요청합니다.  
중간에 톤과 대상이 변경됩니다.

### 3.2 대화 로그
```text
1. 사용자: (회의 메모 30줄 원문 제공) 요약해줘.
2. AI: 확인 질문 3개 — ①공유 대상 ②기한 표기 여부 ③리스크 포함 여부?
3. 사용자: ①팀 내부 ②기한 표기 필요 ③리스크 포함.
4. AI: 4블록 템플릿 출력(결정사항 3 / Action Items 4[담당·기한] / 리스크 2 / 확인필요 1).
5. 사용자: 예산 항목 근거가 어디야?
6. AI: 메모 12줄 "3,000만원"을 인용하고, 그 외 수치는 확인 필요로 분리했다고 설명.
7. 사용자: Action Items에 우선순위 컬럼 추가해줘.
8. AI: 담당·기한·우선순위 3열로 재출력(기존 결정사항 유지).
9. 사용자: (조건 변경) 톤을 경영진 보고용으로 바꾸고, 대상도 임원으로.
10. AI: 톤을 격식체로 전환하고, 요약을 상위 3개 핵심으로 압축. 결정사항·담당·기한은 유지.
11. 사용자: 방금 임원 버전에서 리스크 우선순위도 매겨줘.
12. AI: 리스크에 High/Med/Low를 부여하고, 이전 임원 톤과 구조를 유지.
```

### 3.3 문맥 유지 판정
- 확정된 요구사항은 유지
- 변경된 요구사항만 반영
- 판정: **Pass**

## 4. 심층 인터뷰 대비

### 4.1 무료 모델만 써야 한다면?
정확성이 낮은 무료 모델이라도 다음 방식으로 보완할 수 있습니다.

1. 시스템 프롬프트에 "근거 없으면 확인 필요로 분리" 규칙을 강화합니다.
2. 요약 작업을 블록별로 쪼개서 진행합니다.
3. 출력 후 사실 검증 질문 5개를 자동으로 다시 물어봅니다.

### 4.2 정책이나 수치가 자주 바뀐다면?
- 정책과 수치를 프롬프트에 고정하지 않습니다.
- "제공된 근거 문서 내에서만 인용"하도록 강제합니다.
- 근거 문서를 매번 컨텍스트에 넣습니다.

### 4.3 10턴에서 문맥이 끊긴다면?
- 매 턴 앞에 확정 요구사항 요약을 짧게 다시 넣습니다.
- 예: 톤, 대상, 금지어
- 조건이 바뀌면 그 부분만 갱신합니다.
