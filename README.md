# 2. 시장 분석

## 2.1 시장 환경 요약
본 프로젝트는 한국 PC 시장을 대상으로 하는 성인 코미디 어드벤처 게임이다.  
국내 게임 시장은 모바일 비중이 높지만, PC 플랫폼은 여전히 스팀 기반 인디 게임, 스토리 중심 게임, 퍼즐/어드벤처 장르에서 의미 있는 소비층을 유지하고 있다.

특히 한국 PC 유저 중에는 다음과 같은 세부 수요가 존재한다.

- 고전 포인트 앤 클릭 어드벤처 팬층
- 레트로 감성의 인디 게임 소비층
- 대사와 상황극 중심의 스토리 게임 선호층
- 스트리밍/영상 시청을 통해 게임을 접하는 유저층
- 강한 캐릭터성과 풍자를 선호하는 성인 유저층

따라서 본 프로젝트는 대중형 초대형 타이틀보다는,  
**명확한 취향 타깃을 가진 니치형 PC 게임**으로 보는 것이 적절하다.

## 2.2 플랫폼 적합성
PC 플랫폼은 본 프로젝트와 구조적으로 잘 맞는다.

- 마우스 기반 클릭 조작과 궁합이 좋음
- 장면 전환, 대화창, 인벤토리 UI 구현이 용이함
- 긴 텍스트와 상세한 연출을 부담 없이 수용 가능함
- 스팀 유통을 통한 글로벌/국내 동시 접근이 가능함

Unity 엔진은 아래 이유로 적합하다.

- 2D 중심 어드벤처 제작에 효율적
- UI/상태 전환/씬 관리 구현이 용이함
- 반복 배경, 오브젝트 조합, 대화 시스템 구축에 강점이 있음
- 상대적으로 낮은 사양에서도 안정적 실행이 가능함

즉, 본 프로젝트는 **Unity 기반 PC 어드벤처**로 구현할 때 가장 자연스럽다.

## 2.3 장르 적합성
포인트 앤 클릭 어드벤처는 현재 메인스트림 장르는 아니지만,  
고전 팬층과 인디 취향층을 중심으로 일정한 수요가 유지되고 있다.

본 기획이 장르적으로 가지는 강점은 다음과 같다.

- 강한 캐릭터 서사
- 도시/에피소드 단위 구조
- 아이템 수집과 대화 기반 퍼즐
- 연출 감상이 중요한 구조
- 영상 콘텐츠로도 전달하기 쉬운 상황극 중심 구성

반면 약점도 명확하다.

- 퍼즐 동선이 지나치게 복잡하면 이탈 가능성이 높음
- 현대 유저 기준 템포가 느리면 피로감이 생길 수 있음
- 시대적 유머와 성인 코미디는 한국에서 수용성이 제한적일 수 있음

따라서 본 작품은  
**퍼즐 난이도보다 캐릭터성, 대사, 상황극, 연출의 재미를 우선하는 방향**이 타당하다.

## 2.4 한국 시장에서의 리스크
이 프로젝트는 한국 시장에서 상업적 가능성이 있지만, 다음 리스크를 함께 가진다.

1. 등급 심의 리스크  
   - 성인 유머와 노골적 표현은 등급 상승 요인이 될 수 있음

2. 로컬라이징 난이도  
   - 직역하면 유머가 약해지고, 의역하면 원문의 맛이 변질될 수 있음

3. 대중성 한계  
   - 성인 코미디는 한국 PC 시장에서 메인스트림이 아님

4. 마케팅 제약  
   - 스토어 문구, 썸네일, 홍보 카피에서 표현 수위 제어가 필요함

평가 기준 보강 수정본
본 보강 섹션은 기존 제출물에서 미통과 판정을 받은 항목을 보완하기 위해 추가하였다. 기존 문서의 모델 비교, 업무 과업 입력 템플릿, 페르소나, 실행 로그는 유지하되, 평가자가 지적한 Few-shot 수량, 모호한 입력 처리, 환각 검증 기준, 모델 선정 근거, 심층 인터뷰 대응을 별도 항목으로 정리하였다.
1. 과제 산출물 구분
기존 문서의 후반부는 최종 GDD 본문 비중이 커서, 과제 산출물인 모델 비교 보고서·시스템 설계 문서·실행 로그와 구분이 흐릴 수 있다. 따라서 본 과제의 산출물은 아래 3가지로 구분하여 해석한다.
산출물	내용	기존 문서와의 연결
모델 비교 보고서	Gemini 3 Flash, Claude Code Haiku 4.5, GPT-5.4 mini를 동일 과업에 적용하고 6개 평가 축으로 비교한 결과	문서 초반의 3종 모델 비교와 최종 선정 결론
시스템 설계 문서	AI에게 부여한 페르소나, 업무 과업 입력 템플릿, 금지 조건, 출력 규격, md 변환 규칙	업무 과업 입력 템플릿 및 페르소나/시스템 프롬프트 요약
실행 로그	실제 입력과 응답 흐름을 10턴 이상 기록하여 모델이 조건을 유지하는지 확인한 자료	문서 마지막의 10턴 대화 로그

이 구분을 통해 최종 GDD 본문은 “AI가 생성한 예시 결과물”로 보고, 실제 과제 평가는 모델 비교 보고서, 시스템 설계 문서, 실행 로그를 중심으로 판단하도록 정리하였다.
2. 기능 동작 검증 보강
기능 동작 검증은 AI가 사용자의 업무 과업을 정확히 이해하고, 반복 입력 상황에서도 요구 조건에 맞게 답변하는지 확인하는 절차이다. 본 과업의 기능 동작 검증 목표는 다음과 같다.
• txt 형태의 GDD 문서를 md 문서 구조로 변환할 수 있는가
• 기획 개요, 코어 루프, 핵심 시스템 등 GDD 필수 항목의 누락 여부를 판별할 수 있는가
• 부족한 항목을 보강하되, 원문과 충돌하는 내용을 추가하지 않는가
• 모호한 입력이 들어왔을 때 바로 임의 수정하지 않고 필요한 정보를 되묻는가
• 추가 작성한 내용에 대해 출처, 근거, 또는 추가 이유를 설명할 수 있는가
3. Few-shot 예시 보강
기존 Few-shot은 입력 예시 중심으로 작성되어 있어, 평가 기준에서 요구하는 “예시 3개 이상” 및 “모호한 입력 처리”가 충분히 드러나지 않았다. 따라서 실제 업무에서 사용할 수 있는 입력-기대응답-판정기준 형식으로 보강하였다.
3.1 Few-shot 예시 1: 시장 분석 보강 요청
입력 예시: 해당 GDD 문서에서 수익성이 부족한 것 같아. 시장 분석이 가능하면 시장 분석을 추가하고, 이 게임에 적합한 BM도 제안해줘.
기대 응답: 기존 GDD의 장르, 플랫폼, 타깃 유저를 먼저 유지한 뒤 한국 PC/스팀 시장을 기준으로 시장 분석을 추가한다. BM은 포인트 앤 클릭 어드벤처 장르와 충돌하지 않는 방식으로 제안해야 하며, 과도한 모바일식 과금 구조는 제외한다.
판정 기준: 시장 분석이 기존 기획과 연결되고, BM 제안의 이유와 부적합한 BM 제외 사유가 포함되면 Pass로 판단한다.
3.2 Few-shot 예시 2: 핵심 시스템 보강 요청
입력 예시: 핵심 시스템 부분이 너무 짧아. 경쟁작들의 핵심 시스템을 참고해서 추가할 수 있는 요소가 있는지 분석해줘.
기대 응답: 기존 핵심 시스템을 요약하고 부족한 항목을 식별한 뒤, 경쟁작 시스템을 단순 복사하지 않고 현재 기획에 맞게 변형하여 제안한다.
판정 기준: 기존 장르와 충돌하지 않는 시스템 보강, 적용 이유, 장점과 리스크가 함께 제시되면 Pass로 판단한다.
3.3 Few-shot 예시 3: 코어 루프 세분화 요청
입력 예시: 문서의 코어 루프 부분이 너무 단순해. 실제 플레이 흐름이 보이도록 코어 루프를 더 세분화해줘.
기대 응답: 장소 진입, 오브젝트 탐색, NPC 대화, 단서 획득, 아이템 수집, 아이템 조합, 퍼즐 해결, 사건 진행, 새 장소 해금의 흐름으로 세분화한다.
판정 기준: 단순 순서 나열이 아니라 각 단계에서 플레이어 행동과 보상이 설명되면 Pass로 판단한다.
3.4 Few-shot 예시 4: 모호한 입력에 대한 되묻기
입력 예시: 기획이 별로야. 수정해줘.
기대 응답: “별로”라는 표현은 범위가 넓으므로, 문체 문제인지, 핵심 시스템 문제인지, 시장성 문제인지, 문서 구조 문제인지 먼저 구분 질문을 해야 한다. 사용자가 추가 답변을 하지 않을 경우에는 기본 수정 방향을 제안할 수 있다.
기대 답변 예시:
기획이 별로라고 느껴지는 이유를 먼저 구분해야 할 것 같습니다. 현재 문서에서 수정 방향은 크게 4가지로 나눌 수 있습니다. 1) 문체가 딱딱해서 재미없게 느껴지는 경우, 2) 핵심 시스템이 약해서 게임성이 부족해 보이는 경우, 3) 시장성이나 BM이 부족해 보이는 경우, 4) GDD 구조가 정리되지 않은 경우입니다. 어느 쪽에 가까운지 알려주시면 해당 방향에 맞춰 수정하겠습니다. 별도 선택이 없다면 “문서 구조 정리 + 핵심 시스템 보강 + 시장성 보완” 방향으로 수정하겠습니다.
판정 기준: 모호한 요청을 바로 수행하지 않고 수정 범위를 확인하거나 기본 수정 방향을 제안하면 Pass, 사용자의 의도를 확인하지 않고 전체 문서를 임의로 재작성하면 Fail로 판단한다.
4. 환각 정의 및 창의적 생성과의 구분
본 과업에서 환각은 “AI가 사실과 다른 내용을 실제 정보처럼 작성하거나, 근거가 없는 내용을 확인된 사실처럼 제시하는 오류”로 정의한다. 환각은 단순히 새로운 문장을 생성하는 행위 전체를 의미하지 않는다. GDD 보강 작업에서는 AI가 부족한 항목을 채우고 예시를 제안해야 하므로, 일정 수준의 창의적 생성은 필요한 기능이다.
구분	정의	예시
환각	근거 없는 정보를 사실처럼 작성하는 것	존재하지 않는 시장 통계, 없는 출처, 확인되지 않은 경쟁작 수치를 확정적으로 작성
창의적 생성	사용자의 기획 방향 안에서 새로운 아이디어를 제안하는 것	코어 루프 보강, 퍼즐 예시 추가, UI 항목 제안
추론 기반 보강	원문 내용을 바탕으로 논리적으로 부족한 부분을 채우는 것	PC 플랫폼에 맞는 마우스 조작 UX 보강
과잉 생성	요청 범위를 벗어나 불필요한 설정을 추가하는 것	원문에 없는 세계관, 캐릭터, 스토리를 대량 추가

따라서 본 과업에서는 “부족한 GDD 항목을 보강하는 창의적 생성”은 허용하되, “근거 없는 정보나 원문과 충돌하는 설정을 사실처럼 작성하는 환각”은 오류로 판단한다.
5. 환각 검증 Pass/Fail 판정 기준
환각 검증은 AI가 추가 작성한 내용이 원문 또는 확인 가능한 근거를 바탕으로 작성되었는지 확인하는 과정이다. 본 과업에서는 Pass A, Pass B, Fail의 3단계로 판정한다.
판정	기준
Pass A	원문 내용 또는 명확한 출처에 근거하여 작성되었고, 추가 이유가 분명한 경우
Pass B	직접 출처는 부족하지만 원문 내용에서 논리적으로 추론 가능하며, 추정 표현을 사용한 경우
Fail	근거 없는 정보를 사실처럼 작성하거나, 원문과 충돌하는 내용을 추가한 경우

6. 환각 검증 질문 보강
기존 문서의 환각 검증 질문은 1개뿐이었으므로, 최소 5개 이상 요건을 충족하기 위해 7개의 검증 질문과 기대 정답, 판정 기준을 추가하였다.
6.1 검증 질문 1
질문: AI가 추가한 시장 분석의 출처는 무엇인가?
기대 정답: 시장 수치가 포함된 경우 출처가 있어야 하며, 출처가 없다면 “추정”, “경향”, “가능성”으로 표현해야 한다.
Pass/Fail 기준: 출처 또는 참고 기준이 명확하면 Pass A, 구체 출처는 없지만 추정 표현이면 Pass B, 출처 없는 수치를 확정적으로 작성하면 Fail
6.2 검증 질문 2
질문: 추가된 BM이 기존 장르와 충돌하지 않는가?
기대 정답: 포인트 앤 클릭 어드벤처와 어울리는 BM이어야 하며, 게임 진행을 해치지 않아야 한다.
Pass/Fail 기준: 장르 특성과 연결하여 BM을 제안하면 Pass A, 다소 일반적이지만 충돌하지 않으면 Pass B, 모바일식 과금 등 장르와 충돌하면 Fail
6.3 검증 질문 3
질문: AI가 원문에 없는 캐릭터나 세계관을 임의로 추가했는가?
기대 정답: 원문에 없는 고유명사, 캐릭터, 지역, 세계관은 사실처럼 추가하지 않아야 한다.
Pass/Fail 기준: 원문 범위 안에서만 보강하면 Pass A, 예시로만 제안하면 Pass B, 새 설정을 본문 사실로 추가하면 Fail
6.4 검증 질문 4
질문: 경쟁작 비교가 실제 게임 특성과 맞는가?
기대 정답: 경쟁작의 장르, 핵심 시스템, 시사점이 실제 특성과 어긋나지 않아야 한다.
Pass/Fail 기준: 경쟁작 특성과 비교 내용이 일치하면 Pass A, 일부 일반화는 있으나 큰 오류가 없으면 Pass B, 장르나 시스템을 잘못 설명하면 Fail
6.5 검증 질문 5
질문: GDD 양식 보강이 원문 기획 의도와 충돌하지 않는가?
기대 정답: 기존 기획 방향을 유지하며 부족한 항목만 보완해야 한다.
Pass/Fail 기준: 원문 방향과 일관되면 Pass A, 일부 확장이 있으나 충돌하지 않으면 Pass B, 원문과 반대되는 시스템이나 타깃을 추가하면 Fail
6.6 검증 질문 6
질문: 추가된 기술 요소가 사용 엔진 Unity와 맞는가?
기대 정답: Unity에서 구현 가능한 방식이어야 하며, Unreal 전용 기능처럼 부적절한 내용을 넣으면 안 된다.
Pass/Fail 기준: Unity 기준으로 적절한 시스템을 제안하면 Pass A, 엔진 일반론이지만 적용 가능하면 Pass B, Unreal 전용 내용을 Unity 기능처럼 쓰면 Fail
6.7 검증 질문 7
질문: AI가 불확실한 정보를 확정적으로 표현했는가?
기대 정답: 불확실한 내용은 추정, 제안, 검토 필요 등의 표현으로 구분해야 한다.
Pass/Fail 기준: 불확실성을 명확히 표시하면 Pass A, 일부 단정이 있으나 핵심 오류가 없으면 Pass B, 근거 없는 내용을 확정 사실처럼 표현하면 Fail
7. 환각 검증 실행 예시
검증 예시 1
검증 질문: 추가된 시장 분석에서 한국 PC 시장 관련 수치나 경향에 대한 출처가 있는가?
판정: Pass B
이유: 구체적인 통계 수치보다는 한국 PC/스팀 유저층의 경향을 중심으로 작성되었기 때문에 사실 오류 위험은 낮다. 다만 명확한 외부 출처가 부족하므로 Pass A가 아닌 Pass B로 판정한다.
검증 예시 2
검증 질문: AI가 원문에 없는 캐릭터, 스토리, 지역을 임의로 추가했는가?
판정: Pass A
이유: 최종 출력물은 프로젝트 개요, 시장 분석, 타깃 유저, 디자인 필러, 핵심 시스템 중심으로 작성되었으며, 원문에 없는 고유명사나 새로운 세계관을 임의로 추가하지 않았다.
검증 예시 3
검증 질문: GDD 문서의 부족한 항목을 보강할 때 추가 이유를 설명했는가?
판정: Pass A
이유: 시장 분석, 플랫폼 적합성, 장르 리스크, 힌트 시스템 등은 GDD 완성도와 실제 개발 문서 활용성을 높이기 위한 보강 항목으로 설명되었다.
검증 예시 4
검증 질문: AI가 사용자가 지정한 플랫폼과 엔진을 유지했는가?
판정: Pass A
이유: 목표 시장은 한국, 플랫폼은 PC, 엔진은 Unity로 유지되었고, 시스템 설명도 PC 어드벤처와 Unity 구현에 맞게 작성되었다.
검증 예시 5
검증 질문: AI가 기존 기획과 상반되는 요소를 추가했는가?
판정: Pass B
이유: 대부분 기존 기획과 일치하지만, 일부 시장 분석과 경쟁작 비교는 원문에 직접 있던 내용이 아니라 AI가 보강한 부분이다. 그러나 장르와 플랫폼 방향을 벗어나지는 않았으므로 Fail은 아니다.
8. 모델 선정 근거 보강
최종 모델은 GPT-5.4 mini로 선정하였다. 기존 문서에서는 비용 대비 결과물, 환각 빈도, VS Code 연계성을 중심으로 설명했지만, 평가 기준상 업무 과업 특성과의 연결이 더 필요하다. 따라서 모델 선정 근거를 GDD txt→md 변환 과업과 직접 연결하여 보강하였다.
8.1 GDD 문서 구조 유지 능력
GDD 문서는 기획 개요, 코어 루프, 핵심 시스템, 타깃 유저, 시장 분석 등 여러 항목이 계층적으로 구성된다. GPT-5.4 mini는 10턴 대화 동안 문서 구조를 유지했고, 최종 출력물에서도 md 제목 구조를 안정적으로 유지했다. 이는 txt 원문을 단순 요약하는 것이 아니라, 개발자가 실제로 읽고 수정할 수 있는 md 문서로 변환해야 하는 본 과업에 적합하다.
8.2 긴 문맥 유지 능력
본 업무 과업은 최초 변환 요청 이후 페르소나 추가, 시장/플랫폼/엔진 조건 추가, 시장 조사 보강 요청, 최종 통합 요청이 이어졌다. 따라서 모델은 이전 조건을 잊지 않고 반영해야 한다. GPT-5.4 mini는 10턴 동안 한국 시장, PC 플랫폼, Unity 엔진이라는 조건을 유지했으므로 GDD 문서처럼 여러 조건이 누적되는 작업에 적합했다.
8.3 추가 작성의 통제 가능성
본 과업에서는 부족한 항목을 AI가 보강해야 하지만, 원문과 충돌하는 내용을 추가하면 안 된다. Claude Code Haiku 4.5는 문서 형식과 문맥 유지 능력은 좋았지만, 프롬프트를 지키지 않거나 환각이 발생하는 경우가 있었다. 반면 GPT-5.4 mini는 추가 작성이 비교적 정돈되어 있었고, 원문 기획 방향을 크게 벗어나지 않았다.
8.4 비용 대비 효율
Gemini 3 Flash는 비용과 속도 면에서는 장점이 있었지만, 3회 입력 이후 문맥 유지가 약해졌다. Claude Code Haiku 4.5는 결과물 품질은 높았지만, 10회 사용 비용이 매우 높았다. GPT-5.4 mini는 결과물의 구조, 문맥 유지, 추가 작성 품질, 비용 사이의 균형이 가장 좋았다. 반복 수정이 필요한 GDD 문서 작업에서는 단순히 가장 저렴한 모델보다, 일정 품질을 유지하면서 비용이 과도하게 증가하지 않는 모델이 적합하다.
8.5 md 파일 작업과 개발 환경 연계
본 과업의 최종 결과물은 md 파일 형태로 정리되어야 하며, 이는 VS Code에서 바로 확인하고 수정할 수 있어야 한다. GPT-5.4 mini는 제목, 소제목, 표, 목록 구조를 md 문법에 맞게 안정적으로 출력했기 때문에 후속 편집과 버전 관리가 쉽다. 따라서 VS Code 연계성은 단순히 파일을 열 수 있다는 의미가 아니라, 개발자가 md 문서를 실제 작업 파일로 관리할 수 있는 구조를 제공한다는 점에서 중요하다.
9. 심층 인터뷰 대응
평가에서 지적된 심층 인터뷰 항목을 반영하여, 무료 모델만 사용할 때의 품질 유지 전략, 정책이 자주 바뀔 때의 환각 감소 운영 전략, 10턴 이상에서 문맥이 끊길 때의 프롬프트 구조 개선 방안을 추가하였다.
9.1 무료 모델만 사용할 때 품질 유지 전략
무료 모델만 사용할 경우 모델 성능이 낮거나 문맥 유지가 불안정할 수 있으므로, 한 번에 전체 GDD를 생성하지 않고 작업을 단계별로 분리해야 한다.
1단계: txt 원문을 md 구조로만 변환한다.
2단계: 기획 개요, 코어 루프, 핵심 시스템 등 부족한 GDD 항목만 목록화한다.
3단계: 부족한 항목을 [AI 보강] 표시와 함께 작성한다.
4단계: 추가 작성한 부분에 대해 환각 검증 질문을 수행한다.
5단계: 최종 통합 전 원문과 충돌하는 내용이 있는지 확인한다.
이처럼 업무를 분할하면 무료 모델을 사용하더라도 한 번에 처리해야 하는 정보량이 줄어들어 결과물 품질을 일정 수준 유지할 수 있다.
9.2 정책이나 시장 정보가 자주 바뀔 때 환각 감소 전략
시장 정보, 등급 정책, 플랫폼 정책처럼 자주 바뀌는 정보는 AI가 오래된 정보를 사실처럼 작성할 가능성이 있다. 따라서 최신성이 중요한 항목은 AI가 단정적으로 작성하지 못하도록 제한해야 한다.
• 최신 정보가 필요한 항목은 반드시 출처를 요구한다.
• 출처가 없는 경우 “추정”, “경향”, “검토 필요”로 표시한다.
• 수치 자료는 작성하지 않거나, 작성 시 기준 연도와 출처를 함께 기록한다.
• 정책 관련 내용은 “가능성이 있음”, “확인이 필요함”처럼 확정 표현을 피한다.
• 최종 제출 전 사람이 직접 출처와 정책 변경 여부를 확인한다.
예를 들어 “한국 PC 시장은 성장 중이다”라고 단정하기보다, “한국 PC 인디 시장은 스팀 중심으로 일정 수요가 있는 것으로 볼 수 있으나, 정확한 시장 규모는 최신 자료 확인이 필요하다”라고 작성하는 방식이 더 안전하다.
9.3 10턴 이상에서 문맥이 끊길 때의 프롬프트 구조 개선
10턴 이상 대화가 이어지면 모델이 초반 조건을 잊거나, 사용자가 지정한 시장, 플랫폼, 엔진, 금지 조건을 누락할 수 있다. 이를 방지하기 위해 프롬프트 안에 “고정 조건 블록”과 “현재 작업 블록”을 분리해야 한다.
개선 프롬프트 구조 예시:
[고정 조건]
- 타깃 시장: 한국
- 플랫폼: PC
- 엔진: Unity
- 문서 형식: md
- 역할: 시니어 게임 기획자
- 금지: 원문과 상반되는 내용 추가 금지
- 추가 작성 시 출처 또는 추가 이유 표기

[현재 작업]
- 코어 루프 항목을 세분화한다.
- 기존 내용은 유지한다.
- 부족한 부분만 [AI 보강]으로 표시한다.

[검증 질문]
- 기존 기획과 충돌하는 내용이 있는가?
- 출처 없는 시장 정보가 포함되었는가?
- 사용자가 지정한 플랫폼/엔진 조건을 유지했는가?
이 구조를 사용하면 대화가 길어져도 모델이 핵심 조건을 반복적으로 확인할 수 있다. 특히 GDD처럼 문서가 길어지는 작업에서는 고정 조건과 현재 작업을 분리하는 것이 문맥 유지에 효과적이다.
10. 최종 보강 결론
기존 제출물은 모델 비교와 GDD 변환 결과는 포함하고 있었지만, 기능 동작 검증, 환각 검증, 심층 질문 대응이 부족했다. 본 보강 섹션에서는 다음 내용을 추가하여 평가 기준을 보완하였다.
• Few-shot 예시를 4개로 확장하였다.
• 모호한 입력에 대해 바로 재작성하지 않고 되묻는 케이스를 추가하였다.
• 환각을 “사실 오류”로 정의하고 창의적 생성과 구분하였다.
• Pass A, Pass B, Fail 판정 기준을 명시하였다.
• 환각 검증 질문을 7개로 확장하였다.
• 무료 모델 사용 시 품질 유지 전략을 추가하였다.
• 정책과 시장 정보 변경 시 환각 감소 운영 전략을 추가하였다.
• 10턴 이상 문맥 유지를 위한 프롬프트 구조 개선안을 제시하였다.
• GPT-5.4 mini 선정 근거를 GDD txt→md 변환 과업 특성과 직접 연결하였다.
따라서 수정 후 문서는 단순한 GDD 결과물이 아니라, AI 모델을 특정 업무 과업에 적용하고 성능을 비교하며, 환각과 문맥 유지 문제를 검증한 과제 산출물로 볼 수 있다.
